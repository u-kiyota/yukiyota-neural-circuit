## 1. アーキテクチャの基本方針

**MVCが基本、理解なしに良いソースは作れない**

- 実装のアーキテクチャは様々あるが、MVCが基本
- 他のアーキテクチャも発展系で基本概念にMVCを含んでいる
- エンジニア的には基本中の基本なので、しっかり理解しないと1から良いソースを作れない

**MVCの3原則**

1. **controllerに実装を書かない**
   - entrypointや各処理のメインの関数に実装を入れるべきではない
2. **モデルの責務はファイルを分けてモデルに記載**
   - 例：Orderに対する処理はOrderモデルのファイルを分けてモデルに記載
3. **データ永続化層はrepositoryとしてファイルを分けて実装**
   - 参考：[Qiita - MVCとリポジトリパターン](https://qiita.com/os1ma/items/7a229585ebdd8b7d86c2)

## 2. DB設計レビュー観点

**名前系カラムはNotNull必須**

- `name`系のカラム（`authors`, `publishers` 等）はとくにNotNullが良い。NullだとNothingも入ったデータができてしまう

**statusなどenum系はDBレベルで制限する**

- statusなどenum系は、DBとして保存できる値を制限した方が良いので、enumを使った方が良い（派閥にもよるが原則推奨）

**インデックスの記載と設計**

- インデックスの記載があると良い。検索性能に関わるのでレビューしたい観点
- 全文検索が重要な場合は構成自体を考える必要性がある（elasticsearchを使うかなど）
- 一度デプロイしたDBにインデックスのエラーが出たからと言って、テーブルを消して作り直すのはダメ（本番でデータが消える）。既存リリース状態からできるような形で実装しないといけない

**SystemIDとユーザー表示用コードの分離**

- システムIDとユーザーに表示するコードは分けた方が良い
  - `id`：ULID等でシステム的な一意の値
  - `code`：ユーザーに表示するコード、全ユーザーで一意なカラムを用意する（GitHubのユーザー名のように個人のアカウントを特定できるもの）

**ステータス判断はDBではなくアプリケーションで制御する**

- `is_seidoku` のような「statが精読かどうかの判定」であれば、DBの項目ではなくアプリケーションとして制御した方が良い
- 同じ制御項目に関して、複数箇所で管理を行うと実装が大変になる（片方しか更新しないような変なプログラムを作ってしまったりする）

**データのリレーション設計**

- userに紐づくのであれば、データ構造として `user_booklogs` との紐付きにした方が、メモ数などがわかりやすく、ステータスの判断がしやすい

## 3. コードレビューの考え方

**指摘事項を横串で直す意識を持つ**

- 指摘事項がちゃんと反映されてなかったり、横串修正できてなかったり、しょうもないミスをずっと繰り返していると、レビュアーがだんだん見る気を失ってきて、よいフィードバックもらえなくなる
- 最初のうちに頑張って躾けておくと良い

**PRのFiles changedに余計な差分を入れない**

- PRのFiles changedに関係ない差分が出てしまっている場合は、ブランチ操作がおかしい可能性がある
- 余計なものが入ると、どこが修正点かわからずレビューしづらい
- **PRレビュー依頼前に1ファイルずつ確認して、最終的な差分が意図的か確認する習慣をつけること**

**PR説明をわかりやすく記載する**

- ソースとして特に変なことをしていることはなさそうでLGTMでも、修正の背景・目的・疑問点をPRにわかりやすく記載してもらえるとレビューしやすい
- 「なぜこの修正だけでよいのか」「なぜ共通シナリオ側にも変更が必要なのか」等を記載する

**品質への意識**

- 設計や言われたことを鵜呑みにしてプログラマーになるのではなく、**業務的にこれで回るのかを意識してエンジニアになるようにしてください**
- 懸念点あればまあいっかではなく、相談してください

## 4. ブランチ戦略・デプロイ原則

**バグフィックスのブランチ戦略**

```
main -> release/x.x.x（バグフィックスの場合の版数は3桁目を1つあげる）
release/x.x.x からissueごとの開発ブランチを作成して開発
release/x.x.x にPR
リリース時に release/x.x.x => main にPR、マージ
main => develop に取り込む
```

**本番デプロイの原則：既存リリース状態から変更できること**

- 一度デプロイしたものに破壊的な変更を加えると、devは破壊できるが本番でデプロイできなくなる
  - 例：DBでインデックスのエラーが出たから、一回テーブルを消してインデックスを変えたDBを再作成 → データが消えるので本番では不可能
- 既存のリリース状態からできるような形で実装しないといけない

## 5. テスト設計の考え方

**テスト仕様書の目的と限界の理解**

- テスト仕様書で品質担保できれば完璧だが、実情的には一定以上品質を高める効果は完璧にするのは難しい
- テストする時も「この仕様書を満たせばOK」ではなく、各条件の設定方法が正しいのか、確認方法でその場所だけ見てよいのかを考えながらやると、テスト品質が高まる

**シナリオテストは業務フロー単位で設計する**

- 単体・結合テストに近い記載（中分類での切り）はシナリオテストではない
- **業務フローに沿って一通りが完結することが必要**
  - 例（受発注連携）：見積もり → 受注 → 発注 → 入荷 → 出荷 → 検収 → 月締めがひとまとまり
- 業務として異なるもの（例：現金売上）は同じように全体を記載する
- 単体・結合レベルのバグチェックに近いものは、今の記載（分類）から切り出してチェックでOK（例：存在しない商品コードを手入力して登録）

**テストの種類の使い分け**

- 機能テスト（機能の動作確認）と、シナリオテスト（業務の流れ確認）は別で必要
- 分類はあくまでテストを分ける基準なので特定の決まりはなく、わかりやすい単位で設定すれば良い
- ただしシナリオテストであれば、分類の単位として画面名が来ることはない（業務の流れなので）

## 6. 技術選定・調査の進め方

**技術検証フェーズの設け方**

- 技術検証：ブラウザ操作ツール選定
- 並行して環境構築を進める（Github, docker, terraform, アプリベース作成）

**上流から設計する重要性**

- 上流からやらないと本質的でない
- それを飛ばしてやると（パッケージ導入だけやっても）リスクがある
- データ設計とアーキテクチャ設計が抜けている場合は評価に入れてほしい
- 設計が終わりきっていない場合、一部固まっていない状態で開発に入ることもある。その場合はできそうなところから上手く実装してください

## 7. AIツールの活用

**Claude（AI）の並列開発への活用**

- gitのworktreeを使わなくても、claude標準で並列開発の仕組みを使えるらしい
- 振り返りにAIを壁打ちとして活用するのが良いと考えている（深掘りや具体性が足りていない場合に自分が気がついていない点がわかりそう）
- Claude Codeを使ったAIコーディング：レビューコストが上がることへの対応が必要（コードを書く時間が短縮される分、その時間をコードレビューに回す）

**AI活用時の姿勢**

- FBにAIを使う場合も、記載の文章はすべて自分で読んで確認する
- AIのそのままを使うわけではなく、加筆している
- AIだとFBストレスを感じる受け取り方が起きづらいというメリットもある
- AI側をチューンするより、使う側である自分のドライバを更新することの方が重要（コーディングや設計の本を読むなど）
